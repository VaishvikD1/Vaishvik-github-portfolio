Quality Assurance in Web Development

Research compiled June 2026

Research Question 01

What Does Quality Assurance Mean for Website Development?

Quality Assurance (QA) in web development is the process of systematically testing a website or web application to discover bugs, errors, and oversights before it goes live. It ensures the final product looks and functions exactly as intended — for the client, the end-users, and other stakeholders.

QA activities typically cover five areas: functionality (do all buttons, forms, and links work?), performance (does the site load quickly under load?), usability (is the interface intuitive?), security (are there vulnerabilities?), and compatibility (does it work across browsers and devices?).

A helpful analogy: QA is like the final inspection of a newly built house — all systems are checked and any problems are fixed before the owners move in. Without QA, bugs that slip through can erode user trust and drive visitors away. According to Statista, 60% of customers say website usability is important to them in an online shop, underscoring why QA is a business-critical step, not an optional extra.

Research Question 02

The Two Key Principles of Quality Assurance

Quality Assurance is guided by two foundational principles that shape how teams approach building and delivering digital products:

Fit for Purpose

The product or service must be suitable for its intended use — meeting the specific requirements and needs of the customer or user. In web development, this means the website must actually solve the problem it was built to solve, whether that's selling products, sharing information, or providing a service. It goes beyond technical specifications to address real-world suitability for the planned audience.

Right First Time

All defects and mistakes should be eliminated from the start rather than corrected after the fact. This principle emphasises a quality-first mindset across the entire development team — through preventive planning, thorough testing at every stage, and continuous monitoring. Getting it right from the beginning saves significant time and cost compared to fixing problems after launch.

Research Question 03

How Do Web Standards Relate to Web Quality?

Web standards are technical specifications and guidelines published by the World Wide Web Consortium (W3C) — an international organisation founded in 1994 by Tim Berners-Lee. The W3C's goal is to create standards that keep a consistent level of technical quality and compatibility across the entire web.

Standards like HTML, CSS, and XML describe best practices for web development. By building to these standards, developers ensure their sites work reliably across different browsers and devices — a property known as interoperability. W3C standards are also optimised for security, privacy, web accessibility, and internationalisation.

The connection to quality is direct: following web standards is one of the most important QA practices available. A site built to standards is easier to maintain (no multiple browser-specific code versions), has a longer useful life, and is accessible to more people. Validating HTML and CSS against W3C specifications is a concrete, free step that dramatically improves and ensures a website's quality.

Standards bodies beyond W3C — including WHATWG (HTML Living Standard), TC39/ECMA (JavaScript), and Khronos (WebGL) — also contribute to the ecosystem of quality benchmarks web developers rely on.

Research Question 04

What Does Web Accessibility Mean?

Web accessibility means designing and building websites so that everyone — including people with disabilities — can perceive, understand, navigate, and interact with them. Disabilities that affect web use include visual, auditory, physical, speech, and cognitive conditions.

The internationally accepted benchmark for web accessibility is the Web Content Accessibility Guidelines (WCAG), developed by the W3C's Web Accessibility Initiative (WAI). First published in 1999, and currently at version 2.2 (2023), WCAG organises requirements around four core principles — often called POUR:

Perceivable

Users must be able to perceive all information and UI components through at least one of their senses — for example, providing alt text for images so screen readers can describe them.

Operable

Users must be able to operate all interface elements. For instance, every function navigable by mouse must also be navigable by keyboard alone, for users with motor disabilities.

Understandable

Content and UI behaviour must be understandable — using plain language, consistent navigation, and clear error messages so users know what to do next.

Robust

Content must be built with well-adopted web standards so it works reliably with current and future browsers and assistive technologies like screen readers.

WCAG is not a law itself, but it is the framework referenced by legislation such as the Americans with Disabilities Act (ADA) and many international equivalents. Accessibility is a quality issue as well as an ethical one — an inaccessible site excludes a significant portion of the population.

Personal Website Portfolio (PWP)

Three Steps to Make My PWP Pages More Accessible

Based on this research, here are three concrete steps I will take to improve accessibility on my Personal Web Portfolio, keeping my audience in mind:

  • Add descriptive alt text to every image. Every image on my portfolio — screenshots, project photos, profile picture — will include a meaningful alt attribute that describes what the image shows. This allows screen reader users to understand the visual content, and also helps if images fail to load.

  • Ensure sufficient colour contrast and keyboard navigation. I will use a colour contrast checker (such as the W3C's own tool) to verify that all text meets the WCAG AA minimum ratio of 4.5:1 against its background. I will also tab through every page to confirm all links, buttons, and interactive elements are reachable and usable by keyboard alone — no mouse required.

  • Use proper semantic HTML heading structure. I will organise each page with a single <h1> for the main title, followed by <h2> and <h3> headings in logical order — never skipping levels. This gives screen reader users a clear, navigable outline of each page's content, and also improves SEO as a bonus.
